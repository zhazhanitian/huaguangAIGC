# 敏感词拦截系统：业务流程与设计

本文档描述项目内自建的敏感词拦截系统的业务流程与设计，便于后续替换为阿里云 AI 护栏时做对照与改造。

---

## 一、整体架构

- **词库来源**：`server/src/modules/badwords/sensitive-words.json`（高校级 5000+ 词条），按 `categories` 分类，每个 category 有 `level`（high/medium/low）和 `words` 数组。
- **持久化**：服务启动时 `BadWordsService.onModuleInit` 会从 JSON 解析出所有词，同步到数据库表 `bad_words`（没有则插入，有则按 JSON 的 level 更新）；管理员在后台的增删改也是针对这张表。
- **运行时**：内存缓存 `wordsCache`，TTL 1 分钟，过期后从 DB 重新拉取；实际检测都用这份缓存，避免每次查库。

---

## 二、检测算法（易误触原因）

- **匹配方式**：`badwords.service.ts` 里是**纯子串、不分词**：
  - 把用户内容转小写：`contentLower = content.toLowerCase()`
  - 对每个敏感词转小写：`wordLower = bw.word.toLowerCase()`
  - 判断：`contentLower.includes(wordLower)`
- **问题**：
  - 任意子串命中就触发，例如词库里有「办公室」「第一次」「小姐」「乳头」（医学/正常语境）等，正常描述也会被拦。
  - 没有上下文、没有分词，无法区分“色情/违规”与“正常用语”，误报会很多。
- **级别与动作**：LOW → WARN，MEDIUM → BLOCK，HIGH → BAN；**只有 HIGH 会在后端直接拒绝**（抛 `BadRequestException`），LOW/MEDIUM 不会拦请求，只影响前端是否弹窗/提示。

## 三、业务流程（两条线）

### 3.1 前端预检（仅绘画相关）

- **入口**：`web/src/api/badwords.ts` 的 `checkContent(content)` → `POST /badwords/check`。
- **使用处**：
  - `DrawLayout.vue`：文生图/图生图提交前，对 `prompt + negativePrompt` 调一次 `checkContent`。
  - `CanvasLayout.vue`：节点生成、从提示词面板生成时，同样对拼接后的文案调 `checkContent`。
- **逻辑**：
  - 接口里调用 `checkContent(..., skipLog: true)`，预检一般不记违规日志（HIGH 仍会记）。
  - 前端根据返回的 `highWords / mediumWords / lowWords`：
    - **HIGH**：弹「严重违规」Modal，禁止生成，可选“删除违禁词”；
    - **MEDIUM**：弹窗列出中级敏感词，用户确认后才继续（或删词）；
    - **LOW**：只 toast 提示“包含轻微敏感词，已标记”，不拦。
  - 用户确认继续或删词后，再次点生成会带 `forceGenerate` 等，跳过预检，直接请求后端。

### 3.2 后端强制校验（所有业务统一）

- **入口**：各业务在「创建任务/发消息」前调用 `BadWordsService.assertNoSensitiveWords(content, userId)`。
- **调用点**：

| 业务   | 文件                | 检测内容                         |
|--------|---------------------|----------------------------------|
| 绘画   | `draw.service.ts`   | `prompt + negativePrompt` 拼接   |
| 视频   | `video.service.ts`  | `dto.prompt`                     |
| 音乐   | `music.service.ts`  | 歌词/文本 `textToCheck`          |
| 3D     | `model3d.service.ts`| `dto.prompt`                     |
| 聊天   | `chat.service.ts`   | 消息内容 `content`（两处）       |

- **逻辑**：  
  `assertNoSensitiveWords` 内部调 `checkContent`，**仅当存在 HIGH 级别敏感词时抛 `BadRequestException`**，文案为「内容包含严重违规词汇，禁止生成！此行为已记录。」；MEDIUM/LOW 不抛异常，任务/消息会正常继续。  
  违规记录：`checkContent` 里会对命中词写 `violation_logs` 表（userId、content、matchedWord、action），预检时 skipLog=true 会少记，但 HIGH 仍记。

## 四、数据与后台

- **表**：
  - `bad_words`：id, word, level, category, isActive, createdAt
  - `violation_logs`：id, userId, content, matchedWord, action, createdAt
- **管理端**：`admin` 下「敏感词管理」`BadWordsManage.vue`，敏感词列表/增删改/统计、违规记录列表；接口在 `badwords.controller.ts`（/badwords/word、words、stats、violations 等）。

## 五、替换成阿里云 AI 护栏时的改造要点

1. **后端**  
   - 用阿里云内容安全/AI 护栏 API 替代当前的「遍历词库 + includes」逻辑。  
   - 可保留 `BadWordsService` 的**接口形状**（例如仍提供 `checkContent`、`assertNoSensitiveWords`），内部改为调阿里云，返回结构尽量兼容现有的 `CheckContentResult`（或做一层映射），这样业务代码改动最小。  
   - 若阿里云返回的是「通过/不通过 + 风险标签」，需要定一个映射规则：哪种标签视为 HIGH（直接拒绝）、哪种仅记录或仅前端提示。

2. **前端**  
   - 绘画的预检：继续调同一个「检测接口」即可（例如仍是 `POST /badwords/check`），只要后端把阿里云结果转成现有的 `passed/canGenerate/needConfirm/hasWarning` 和 `highWords/mediumWords/lowWords`，前端逻辑可以基本不动；若阿里云不返回具体词，可用「风险类型」代替展示文案。

3. **词库与违规记录**  
   - 若完全切到阿里云，`sensitive-words.json` 和 `bad_words` 表可以不再参与检测，但可保留表和管理端用于「自定义黑名单」或与阿里云结果叠加（看产品需求）。  
   - `violation_logs` 可继续用，写入时把 `matchedWord` 改为阿里返回的标签或摘要即可。

4. **模块与依赖**  
   - `BadWordsModule` 仍被 Draw/Video/Music/Chat/Model3d 引用，只需在 `BadWordsService` 内换实现；若阿里云需要配置（如 region、accessKey），在模块或配置里注入即可。

## 七、已替换为阿里云内容安全（AI 护栏）

当前项目已**去掉敏感词系统入口**，改为使用**阿里云内容安全（AI 护栏）**进行文本与图片审核。

### 7.1 后端

- **模块**：`server/src/modules/content-moderation/`
- **服务**：`ContentModerationService` 提供 `checkText`、`checkImage`、`assertTextSafe`、`assertImageSafe`
- **接口**：
  - `POST /content-moderation/check-text`  body: `{ content }`  → 前端预检
  - `POST /content-moderation/check-image` body: `{ imageUrl }` → 前端预检（须公网 URL）
- **环境变量**（`server/.env`）：
  - `ALIYUN_CONTENT_MODERATION_ACCESS_KEY_ID` / `ALIYUN_CONTENT_MODERATION_ACCESS_KEY_SECRET`（或 `ALIYUN_AI_GUARD_*`）
  - 可选：`ALIYUN_CONTENT_MODERATION_ENDPOINT`（默认 `green-cip.cn-shanghai.aliyuncs.com`）
  - 可选：`CONTENT_MODERATION_IMAGE_STRICT=true` 开启图片严格模式（仅当所有检测维度均为 none 时才通过）
  - 本地上传图片审核需公网可访问：`SITE_PUBLIC_URL=https://你的域名`
- **接入点**：绘画、视频、音乐、聊天、3D 创建任务前调用 `assertTextSafe`；绘画/视频/3D 的参考图及上传接口调用 `assertImageSafe`。
- **图片审核偏严**：若与第三方（如豆包）相比我方通过但第三方拦截，可在 [AI 安全护栏控制台](https://yundun.console.aliyun.com/?p=guardrail) → 防护配置 → 检测项配置，将 **AIGC 输入图片** 的防护等级设为「高」，并适当调高敏感度。

### 7.2 前端

- **API**：`web/src/api/content-moderation.ts` 的 `checkText`、`checkImage`
- **预检**：`DrawLayout.vue`、`CanvasLayout.vue` 在提交前调用 `checkText`，不通过则弹「内容安全提示」并阻止提交。

### 7.3 管理后台

- 敏感词管理菜单与路由已移除（`/badwords`、BadWordsManage 入口已去掉）。
